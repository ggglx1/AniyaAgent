# AniyaAgent 产品计划

## 1. 产品定位

AniyaAgent 的目标是从个人 Agent demo 演进为一个参考 CowAgent 形态的个人多通道通用 Agent 平台。

它不应该被定位为纯 Coding Agent。Coding 是重要能力之一，但产品中心应当放在个人生活、个人任务、日常工作流、长期记忆和多入口使用体验上。

核心定位：

> 一个可以从 Web、手机、即时通讯、定时任务等多个入口访问的个人 Agent，能够记住用户长期上下文，管理任务、提醒、知识和日常工作流，并在安全边界内调用本地能力完成实际事情。

参考 CowAgent 后，AniyaAgent 应向这个方向靠拢：

- 多通道入口，而不是单一 Web/CLI。
- 统一 Agent 大脑，而不是每个入口各自一套逻辑。
- 记忆、任务、技能、调度、控制台共同组成个人工作台。
- Coding 能力作为工具能力融入个人工作流，而不是产品唯一重心。

## 2. 当前现状

AniyaAgent 当前已经具备较强的 Agent 内核基础：

- 本地 Agent 循环。
- 工具调用能力。
- 文件、Shell、任务、计划、后台任务等本地执行能力。
- 记忆系统。
- 技能系统。
- 权限确认。
- 手机 Web 访问入口。
- 多 Agent / 子任务雏形。
- 测试与 benchmark 基础。

但从 CowAgent 的角度看，当前 AniyaAgent 更像“Agent runtime + Web bridge”，还不是完整的“多通道个人 Agent 平台”。

主要差距：

- 缺少类似 CowAgent 的统一通道模型。
- Web 入口还是特殊入口，没有被当作通道体系中的一个成员。
- 外部消息平台尚未接入。
- 定时任务还没有和通道投递、个人提醒形成完整闭环。
- 记忆、任务、技能、日志、调度等能力尚未形成统一产品控制台。
- 产品叙事仍偏 Coding/执行，还需要转向个人生活与工作流。

## 3. 产品原则

### 3.1 参考 CowAgent：先统一通道，再扩展平台

CowAgent 的核心优势之一是把 Web、终端、微信、飞书、钉钉、企微、Telegram、Slack 等入口统一成 Channel。

AniyaAgent 也应采用同样的产品思路：所有入口只是不同通道，通道负责接收消息和发送回复，真正的 Agent 逻辑保持统一。

### 3.2 一个 Agent 大脑，多个入口

用户不应该感知到“Web Agent”“手机 Agent”“IM Agent”是不同系统。

不同入口可以有不同交互形式，但它们应共享：

- 记忆。
- 会话。
- 任务。
- 提醒。
- 技能。
- 工具能力。
- 权限策略。
- 审计记录。

### 3.3 个人工作流优先

参考 CowAgent 的产品方向，AniyaAgent 应从“能执行工具”升级为“能帮助用户持续处理个人事务”。

优先场景：

- 每日计划。
- 任务拆解和跟进。
- 定时提醒。
- 项目追踪。
- 知识沉淀。
- 文件和资料整理。
- 个人复盘。
- 长期目标管理。

### 3.4 Web 控制台是默认管理入口

CowAgent 的 Web 控制台不只是聊天界面，也承担模型、通道、技能、记忆、任务等管理能力。

AniyaAgent 也应把 Web/mobile 入口升级为个人 Agent 控制中心，而不是只保留一个简单聊天页。

### 3.5 本地执行能力保持为差异点

CowAgent 更偏多通道平台，AniyaAgent 当前更强的是本地执行和任务 runtime。

未来产品应把两者结合：

- 参考 CowAgent 的多通道入口。
- 保留 AniyaAgent 的本地执行、权限、任务和测试基础。
- 形成“多通道个人入口 + 本地执行 Agent”的组合优势。

## 4. 目标架构

目标架构参考 CowAgent 的分层思想，但用于 AniyaAgent 的个人 Agent 方向。

```text
用户入口
  - Web / Mobile
  - CLI
  - 即时通讯平台
  - 定时任务
  - 未来的邮件、Webhook、桌面端

        |
        v

通道层
  - 每个入口独立接入
  - 消息统一成内部请求
  - 回复转换成通道可展示的形式
  - 通道生命周期可管理

        |
        v

统一 Agent Runtime
  - 会话管理
  - 队列与并发控制
  - 取消任务
  - 审计记录
  - 上下文管理

        |
        v

Agent 能力层
  - 模型调用
  - 工具调用
  - 记忆
  - 任务
  - 调度
  - 技能
  - 权限
  - 多 Agent 协作

        |
        v

投递层
  - 文本回复
  - 文件 / 图片 / 语音
  - 主动通知
  - 权限确认
  - 定时任务结果
```

## 5. 核心产品模块

### 5.1 多通道体系

参考 CowAgent，通道体系应成为 AniyaAgent 的第一层产品能力。

通道职责：

- 接收平台消息。
- 识别用户、群聊、会话。
- 把平台消息转成统一请求。
- 把 Agent 回复转成平台消息。
- 处理平台特有的文件、图片、语音、长文本和状态。

优先通道顺序：

- Web/mobile：默认控制台和主入口。
- CLI：本地调试和高级用户入口。
- 定时任务：系统主动触发入口。
- 一个低复杂度外部通道：用于验证多通道闭环。
- 飞书/微信/企微：面向中文个人工作流的高价值入口。

### 5.2 Web 控制中心

参考 CowAgent Web 控制台，AniyaAgent 的 Web/mobile 应承担统一管理职责。

目标页面：

- Chat：对话入口。
- Tasks：个人任务。
- Schedule：提醒和定时任务。
- Memory：长期记忆。
- Skills：技能管理。
- Channels：通道管理。
- Logs：运行记录和权限记录。
- Settings：模型、权限、安全、工作区配置。

Web 控制中心的定位是“个人 Agent 仪表盘”，不是简单聊天窗口。

### 5.3 个人任务和工作流

AniyaAgent 要从“能完成一次请求”升级为“能持续管理用户事务”。

参考 CowAgent 的任务、记忆、调度、技能组合，AniyaAgent 应重点支持：

- 把对话中的事项沉淀为任务。
- 把任务安排到未来提醒。
- 根据记忆自动补全上下文。
- 根据项目状态生成下一步。
- 定期主动复盘。
- 把文件、资料、网页、聊天记录整理为知识。

### 5.4 记忆与知识

CowAgent 的价值不只在通道，也在长期上下文。

AniyaAgent 的记忆系统应服务于个人成长和个人事务管理。

记忆应覆盖：

- 用户偏好。
- 长期目标。
- 当前项目。
- 人际关系。
- 常用流程。
- 重要决策。
- 用户对 Agent 的反馈。

记忆产品原则：

- 用户可见。
- 用户可编辑。
- 用户可撤销。
- Agent 使用关键记忆时应可解释。

### 5.5 调度与主动触达

参考 CowAgent 的定时任务与多通道能力，AniyaAgent 不应只被动回答，还应能主动触达用户。

典型场景：

- 每天早上推送今日计划。
- 每晚生成复盘。
- 到时间提醒跟进某人。
- 定期检查项目状态。
- 定期总结知识库。
- 在任务到期前提醒。

调度能力必须和通道能力结合：任务执行完以后，结果应该能投递到用户选择的入口。

### 5.6 安全与信任

AniyaAgent 的本地执行能力更强，因此安全比普通聊天 Agent 更重要。

参考 CowAgent 的多通道形态后，还需要增加通道信任分层：

- 本地 Web/CLI 可以承担高信任操作。
- 外部 IM 默认只能触发低风险任务。
- 高风险工具需要用户确认。
- 权限确认应清晰展示原因和影响。
- 所有关键动作应可审计。

## 6. MVP 路线图

### Phase 0：产品方向对齐

目标：把 AniyaAgent 的目标从“本地 Coding Agent demo”明确调整为“参考 CowAgent 的个人多通道 Agent 平台”。

结果：

- 产品定位明确。
- CowAgent 作为主要参考对象。
- 当前能力与目标能力差距清晰。
- 后续工作围绕通道、控制台、个人工作流展开。

### Phase 1：Web 入口产品化

目标：把 Web/mobile 从简单聊天入口升级为个人控制中心。

参考 CowAgent：

- Web 是默认入口。
- Web 管理配置和能力。
- Web 管理通道。
- Web 展示任务、记忆、技能、运行状态。

结果：

- 用户可以通过 Web 理解和管理自己的 Agent。
- Web 不再只是发送 prompt 的页面。

### Phase 2：统一通道模型

目标：让 AniyaAgent 的所有入口都遵循 CowAgent 式 Channel 思路。

结果：

- Web、CLI、定时任务、外部平台都被视为通道。
- 各通道共享 Agent runtime。
- 新通道接入路径清晰。
- 通道可以启动、停止、配置和观察状态。

### Phase 3：外部通道验证

目标：接入第一个真实外部通道，验证 AniyaAgent 作为多通道 Agent 平台的可行性。

建议优先级：

- 先选简单、稳定、调试成本低的通道。
- 再接入飞书、微信、企微等高价值中文场景。

结果：

- 用户可以离开 Web，从日常沟通入口调用 AniyaAgent。
- 多通道会话、任务、记忆可以共享。

### Phase 4：个人工作流成型

目标：让 AniyaAgent 从“可聊天、可执行”变成“能持续帮助用户生活和工作”。

结果：

- 用户可以自然创建任务和提醒。
- Agent 能做每日计划和复盘。
- 记忆能持续影响回答和任务管理。
- 定时任务能主动投递结果。
- Web 控制台能展示完整个人工作流。

### Phase 5：长期个人平台

目标：让 AniyaAgent 成为长期运行的个人 Agent 平台。

结果：

- 多通道稳定。
- 记忆可信。
- 任务和提醒可靠。
- 权限和审计清晰。
- 用户能长期依赖它管理个人事务。

## 7. 近期产品优先级

不写具体实现，近期优先级应按产品价值排序：

1. 明确采用 CowAgent 式多通道平台方向。
2. 把 Web/mobile 定义为默认控制中心。
3. 把通道作为统一入口模型。
4. 把任务、记忆、调度、技能整合成个人工作台。
5. 先验证一个外部通道，再扩展高价值中文通道。
6. 保留本地执行和权限控制作为 AniyaAgent 的差异点。

## 8. 产品指标

衡量 AniyaAgent 是否向 CowAgent 式个人平台演进，可以观察：

- 用户是否从多个入口使用 Agent。
- 用户是否持续创建和完成任务。
- 定时任务是否能可靠触达用户。
- 记忆是否被用户检查、修正和信任。
- Web 控制台是否成为主要管理入口。
- 外部通道是否真正承载日常工作流。
- Coding 请求占比是否下降到工具能力层，而不是唯一使用场景。

## 9. 关键风险

- 只复制 CowAgent 的通道数量，没有形成统一产品体验。
- 过早接入复杂通道，导致核心模型不稳定。
- Web 控制台能力不足，用户无法管理记忆、任务和通道。
- 本地执行权限过强，外部通道触发风险不可控。
- 记忆不透明，用户不信任 Agent。
- 产品继续停留在 Coding demo，没有转向个人工作流。

## 10. 战略建议

AniyaAgent 应参考 CowAgent，但不要简单复制 CowAgent。

CowAgent 值得参考的是：

- 多通道抽象。
- Web 控制台。
- 统一消息处理。
- 记忆、任务、技能、调度组合。
- 面向个人日常入口的产品方向。

AniyaAgent 应保留自己的优势：

- 本地执行能力。
- 权限控制。
- 任务 runtime。
- 测试和 benchmark。
- 面向个人深度工作流的可扩展性。

最终目标：

> 做一个参考 CowAgent 形态，但更强调本地执行、安全控制和个人长期工作流的多通道个人 Agent 平台。
